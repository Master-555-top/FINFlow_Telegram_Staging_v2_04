## v1.7 — AI Memory + Learning
Добавлены:
- `lib/memoryBrain.ts` — сбор полного контекста для ИИ: текущая дата/время, фонды, смены, заказы, расходы, сон, корректировки.
- `supabase/ai_learning.sql` — память ИИ, история вопросов и журнал исправлений.
- `app/api/ai-memory/route.ts` — быстрый API-роут для контекстных ответов карточками.
Как должен работать умный режим:
1. Пользователь исправляет смену, расход или фонд.
2. Изменение сохраняется в базу и в `data_corrections`.
3. Realtime обновляет дашборд.
4. ИИ получает новый контекст и больше не использует старую ошибочную цифру.
5. Вопросы вроде «Можно ли букет за 5 000₽ сегодня?» считаются через текущее время, похожие дни, фонды, дедлайны и историю трат.
Для настоящего AI подключи `OPENAI_API_KEY` и замени fallback в `/api/ai-memory/route.ts` на вызов OpenAI Structured Outputs.
## v1.8 — Статистика в каждом разделе, кэш, бессрочная история
Добавлено:
- отдельная логичная статистика в каждой вкладке: Главная, Смена, Расходы, Фонды, ИИ, История, Система;
- локальный кэш `localStorage` с версией `finflow-v1.8`, чтобы приложение открывалось быстро даже до ответа Supabase;
- Supabase Realtime: после изменений интерфейс подтягивает актуальные данные;
- бессрочное хранение: данные не удаляются физически;
- soft-delete: запись получает `deleted_at` и пропадает из аналитики, но остаётся в базе;
- audit log: insert/update/soft_delete фиксируются в `audit_log`;
- `section_cache`: заготовка для кэшированных агрегатов по разделам и периодам;
- `data_versions`: журнал версий данных для будущего восстановления и AI-памяти.
### SQL для v1.8
После базовой схемы выполни:
```sql
supabase/retention_cache_audit.sql
```
### Правило данных
В FinFlow ничего не удаляется просто так. Любая правка — это новая версия состояния и запись в журнале. Аналитика считает только активные строки (`deleted_at is null`), но история сохраняется бессрочно.
### Как работает скорость
1. Интерфейс сначала берёт данные из локального кэша.
2. Потом подтягивает актуальную Supabase-версию.
3. Realtime слушает изменения и обновляет разделы.

# FinFlow Project Memory
## Цель проекта
FinFlow — личная операционная система дня, работы, денег, сна, фондов, решений и роста.
Приложение должно отвечать каждый день:
- что уже сделано;
- что осталось;
- что нельзя трогать;
- что можно потратить;
- сколько нужно заработать;
- как не провалить дедлайны;
- как вырасти без выгорания;
- что делать завтра.
## Визуальная база
Канонический стиль: v2.0 UI Safe.
- тёмная фиолетово-синяя палитра;
- без белых засветов поверх текста;
- без чужих серых блоков;
- readable cards;
- safe area;
- clamp-типографика;
- контрастный текст;
- суммы не вылезают;
- кнопки не перекрываются.
## Контекст времени
На ключевых экранах всегда показывать:
- дата;
- день недели;
- текущее время;
- время суток: ночь / утро / день / вечер.
Это нужно ИИ и пользователю для контекстных решений: поздно ли работать, реально ли купить сегодня, какие дни похожи, что делать со сном.
## Day Core
Главный объект системы — день:
сон → смена → доход → расход → чистый доход → фонды → цели → завтра.
Разделы остаются, но являются деталями дня, а не разрозненными экранами.
## Денежные правила
- 8 500₽ чистыми в день — базовая дневная цель.
- 11 000₽ грязными — базовая грязная цель.
- 59 500₽ чистыми в неделю.
- 212 500₽ чистыми в месяц.
- Машина: 45 000₽ к каждому 6 числу.
- Банк/банкротство: 15 000₽ к каждому 15 числу.
- Встречи: держать 3 000₽.
- Рабочий фонд: защищён, минимум 2 500₽, желательно 7–10k.
## Adaptive Growth
Рост адаптивный:
- если прогресс стабильный — можно повышать цель;
- если сон, фонды или работа просели — не гонимся за 300 000₽;
- сначала восстановление режима, обязательств и рабочего фонда;
- 250k/300k — уровни роста, а не давление.
CORE: День = сон → смена → доход → расход → чистый доход → фонды → цели
```

# Full Editing Spec
Все сущности должны редактироваться и soft-delete/restore без потери истории.
## Обязательные сущности
- transactions: доходы, расходы, фондовые операции
- work_shifts: смены
- shift_rides: заказы внутри смены
- funds: фонды и дедлайны

3. Восстановление = `deleted_at = null` + audit action `restore`.
4. Импортированные записи показывают `source_raw_text`, чтобы можно было сверить с оригиналом.
